## Platform

disabilitywiki.org is a **static [Astro Starlight](https://starlight.astro.build/)
build** hosted on **Cloudflare Pages** (project `disability-wiki`).

- **Root directory**: `site/`
- **Deploys**: automatically on merge to `main` (a Pages build runs in ~2 min).
- **Publishing = merging to `main`.** There is no separate publish step and no human
  review gate beyond CI — a merge reaches the live site within one build cycle.
- **Preview**: https://disability-wiki.pages.dev, plus a per-branch preview deploy on
  every PR.
- **DNS / cache / TLS**: Cloudflare, zone `disabilitywiki.org`. Apex + `www` are
  proxied CNAMEs to `disability-wiki.pages.dev`. `/en/*` paths 301 to `/*` (the EN
  locale is served at the root).

### Local development

```bash
cd site
npm ci
npm run dev      # local dev server (astro dev)
npm run build    # production build → site/dist (astro build + index-redirects + service worker)
```

The build compiles all wiki content (~540 pages) in seconds. If `npm run build`
succeeds locally, the Cloudflare Pages build will too — it runs the same command.

## How content maps to the site

Content markdown lives in the **repo root**, organized by category directory
(`benefits/`, `rights/`, `crisis/`, `housing/`, …). Spanish pages live under `es/`
(e.g. `es/benefits/us/ssi.md`). These directories are **symlinked** into
`site/src/content/docs/`, so the static build reads the same files that are the
repo's source of truth — no content is copied or moved.

- URL mapping: `benefits/us/ssi.md` → `/benefits/us/ssi` (EN at root),
  `es/benefits/us/ssi.md` → `/es/benefits/us/ssi`.
- **Sidebar** is autogenerated from the content directories (ordered
  start/foundations/crisis first) — do **not** hand-maintain a nav list.
- **Redirects**: `site/public/_redirects` (Cloudflare `_redirects` format);
  index-page redirects are generated at build by `site/tools/gen-index-redirects.mjs`.
- **PWA / offline**: the site is installable (`site/public/manifest.webmanifest`)
  and works offline. `site/tools/gen-sw.mjs` generates the service worker at build:
  all `crisis/` + `es/crisis/` pages are precached (discovered by walking `dist/`,
  never hand-listed); HTML is **network-first** so a stale crisis number is never
  served while online. `site/public/_headers` keeps `sw.js` out of the edge cache.
- Because the build only includes what's symlinked under `site/src/content/docs/`,
  non-content files (`.claude/`, `docs/`, root `*.md`) are **not** published — unlike
  the Wiki.js era, there is no leak-and-sweep problem.

To edit, create, delete, or publish content, use the **disability-wiki-edit** skill;
for house voice/structure use **disability-wiki-page**. For Spanish, use
**spanish-wiki-translation**.

## CI merge-gate

Every PR runs [`.github/workflows/ci.yml`](.github/workflows/ci.yml):

- **Astro build** (blocking) — the same build Cloudflare runs.
- **`scripts/validate_wiki_links.py --strict`** (blocking) — baseline is 0 broken
  internal links.
- **`scripts/check_accessibility.py`** (advisory) — content a11y; current findings
  are the intentional emoji-as-info house style.

Because publishing is merge-to-main with no review gate, CI is the safety net. A red
check on a PR is the early warning that a merge would break the live site.

## Operations

- **Incident response** (site down, wrong life-safety content, bad merge):
  [`docs/INCIDENT_RESPONSE.md`](docs/INCIDENT_RESPONSE.md). Rollback is
  `git revert` + redeploy, or a Cloudflare Pages dashboard deployment-rollback.
- **Content accuracy** (crisis numbers, benefits figures, legal deadlines):
  the **disability-wiki-accuracy** skill and [`docs/CLAIMS.md`](docs/CLAIMS.md).
- **Cloudflare caches rendered HTML.** After a fix, the edge can still serve the old
  page; a `?v=` query param does **not** bust it. Verify against the built source or
  purge the URL in the Cloudflare dashboard (Caching → Purge).
